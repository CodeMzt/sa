# 嵌入式C语言开发工作流

## 概述
本工作流专为嵌入式C语言项目（基于Renesas RA微控制器）设计，涵盖代码开发、测试、审查和提交全过程。工作流强调代码质量、安全性和可维护性。

## 工作流程

### 阶段1：规划与分析（Plan编写）

#### 1.1 明确开发目的
- 分析用户需求，明确本次开发的具体目标
- 确定功能范围、性能要求和约束条件
- 识别与现有系统的集成点

#### 1.2 任务分解
- 将复杂任务拆分为独立的子任务
- 为每个子任务定义清晰的输入输出
- 评估各子任务的技术难点和风险
- 创建优先级排序

#### 1.3 技术调研
- 查阅相关硬件文档（Renesas RA系列手册）
- 分析现有代码库中的类似实现
- 确定所需的硬件接口和驱动配置

### 阶段2：Prompt审查与计划完善

#### 2.1 重新审视Prompt
- 仔细阅读用户提供的所有需求说明
- 检查是否有歧义或不完整的地方
- 识别隐含需求和技术约束

#### 2.2 计划验证
- 验证任务分解是否覆盖所有需求
- 检查技术可行性
- 评估时间估计的合理性
- 如有必要，调整计划并重新与用户确认

### 阶段3：版本控制准备

#### 3.1 仓库检查
```bash
# 检查当前是否在git仓库中
git status

# 如果不是，询问用户是否需要创建仓库
# 用户确认后，执行：
git init
```

#### 3.2 分支管理
```bash
# 创建新的功能分支
git checkout -b feature/[简短描述]

# 或者修复分支
git checkout -b fix/[问题描述]
```

### 阶段4：第一轮迭代开发

#### 4.1 文件结构准备
- 根据功能需求确定文件存放位置
- 嵌入式项目文件结构：
  - `src/modules/[模块名]/` - 硬件驱动和模块代码
  - `src/tools/` - 通用工具函数
  - `ra_gen/` - 生成的配置代码（通常不修改）
  - `ra_cfg/` - 配置头文件

#### 4.2 代码编写原则
1. **逐步实现**：先实现核心功能，再添加边缘情况处理
2. **模块化**：每个函数尽量简短，功能单一
3. **实时性考虑**：避免阻塞调用，合理使用RTOS特性
4. **资源管理**：注意内存分配和硬件资源使用

#### 4.3 编写过程
- 按照计划中的子任务顺序进行
- 每个子任务完成后进行初步测试
- 编写量大的任务拆分为多个提交

### 阶段5：代码审查与迭代优化（3-4轮迭代）

#### 5.1 第一轮审查：逻辑正确性
- **指针安全**：检查所有指针的初始化和空值检查
- **数组边界**：验证所有数组访问都在有效范围内
- **硬件寄存器**：确保正确配置和访问硬件寄存器
- **中断处理**：验证中断服务例程的快速执行和非阻塞特性

#### 5.2 常见嵌入式Bug检查点
1. **内存泄漏**：动态分配的内存在适当位置释放
2. **竞态条件**：多任务访问共享资源时的同步机制
3. **栈溢出**：评估函数调用深度和局部变量大小
4. **硬件时序**：验证通信接口的时序要求（SPI、I2C、UART）
5. **电源管理**：低功耗模式下的外设状态管理

#### 5.3 第二轮审查：性能优化
- 减少不必要的内存拷贝
- 优化循环和条件判断
- 使用查表法替代复杂计算
- 合理使用inline函数减少调用开销

#### 5.4 第三轮审查：可维护性
- 添加充分的注释，特别是硬件相关操作
- 确保函数和变量命名清晰
- 模块间接口定义明确
- 错误处理机制完善

#### 5.5 第四轮审查：测试覆盖
- 验证核心功能的单元测试
- 集成测试验证模块间协作
- 边界条件测试
- 错误恢复测试

### 阶段6：代码风格与质量检查

#### 6.1 命名规范
- **变量和函数**：小写字母+下划线（snake_case）
  - 正确：`motor_speed`, `calculate_pid_output()`
  - 错误：`MotorSpeed`, `calculatePIDOutput()`
- **常量**：全大写+下划线
  - 正确：`MAX_BUFFER_SIZE`, `CAN_BAUD_RATE`
- **类型定义**：`_t`后缀
  - 正确：`motor_config_t`, `can_frame_t`

#### 6.2 代码格式
- 使用4个空格缩进，不使用Tab
- 函数之间用空行分隔
- 操作符两侧加空格
- 每行不超过80个字符（嵌入式项目建议）

#### 6.3 头文件规范
```c
// 防止重复包含的保护宏
#ifndef MODULE_NAME_H
#define MODULE_NAME_H

// 包含必要的系统头文件
#include <stdint.h>
#include <stdbool.h>

// 类型定义
typedef struct {
    uint32_t param1;
    uint16_t param2;
} module_config_t;

// 函数声明
bool module_init(module_config_t *config);
void module_process(void);

#endif // MODULE_NAME_H
```

#### 6.4 源文件规范
```c
// 文件头部注释
/**
 * @file module_name.c
 * @brief 模块功能描述
 * @author [开发者]
 * @date [创建日期]
 * 
 * @details 详细功能描述
 * 硬件依赖：[硬件模块]
 * 资源使用：[内存、外设等]
 */

#include "module_name.h"

// 静态函数（仅在本文件内使用）
static void internal_function(void) {
    // 实现
}

// 公有函数实现
bool module_init(module_config_t *config) {
    if (config == NULL) {
        return false;  // 参数检查
    }
    // 初始化代码
    return true;
}
```

#### 8.2 Review重点

3. **资源使用**：内存、堆栈、外设
4. **错误处理**：异常情况的恢复机制
5. **API设计**：模块接口的易用性和一致性

## 特殊考虑：嵌入式项目特点

### 硬件抽象层
- 驱动代码应与硬件细节分离
- 提供统一的接口给应用层
- 方便硬件平台迁移

### 实时性约束
- 中断服务例程（ISR）尽可能简短
- 避免在ISR中进行复杂计算或阻塞调用
- 使用RTOS任务间通信机制

### 资源限制
- 内存使用优化
- 避免动态内存分配（如必须，使用静态池）
- 外设资源共享管理

### 低功耗设计
- 合理使用休眠模式
- 外设按需启用
- 时钟频率动态调整

## 工具链集成

### 编译检查
- 确保代码能通过项目编译系统
- 处理所有警告（将警告视为错误）

### 静态分析
- 使用静态分析工具检查潜在问题
- 特别关注指针、数组和并发问题

## 应急处理

### 遇到技术难题
1. 首先查阅硬件文档和现有代码
2. 分析类似模块的实现
3. 如仍无法解决，记录问题并询问人工

### 计划变更
- 如发现原计划不可行，及时调整
- 与用户沟通变更内容和原因
- 更新任务分解和优先级
