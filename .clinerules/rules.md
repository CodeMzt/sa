# Cline 开发规则与规范

## 项目概述
本项目是一个基于 Renesas RA 微控制器的嵌入式系统，包含以下核心功能：
- CAN FD 通信与电机控制
- WiFi 连接与网络通信
- 触摸屏显示与用户界面（基于 LVGL）
- 语音识别（基于 Edge Impulse）
- FreeRTOS 实时操作系统

## 代码风格规范

### 命名约定
1. **变量和函数名**：小写字母 + 下划线（snake_case）
   - 正确：`motor_speed`, `can_frame_buffer`, `calculate_pid()`
   - 错误：`MotorSpeed`, `canFrameBuffer`, `calculatePID()`

2. **常量和宏**：全大写 + 下划线
   - 正确：`MAX_CAN_PAYLOAD`, `WIFI_RETRY_COUNT`, `SCREEN_WIDTH`
   - 错误：`maxCanPayload`, `WifiRetryCount`

3. **类型定义**：使用 `_t` 后缀
   - 正确：`motor_config_t`, `can_message_t`, `wifi_state_t`
   - 错误：`MotorConfig`, `CANMessage`, `WifiState`

4. **文件命名**：小写字母 + 下划线
   - 正确：`robstride_motor.c`, `drv_canfd.h`, `nvm_manager.c`
   - 错误：`RobstrideMotor.c`, `DrvCANFD.h`

### 代码格式
1. **缩进**：使用 4 个空格，禁止使用 Tab
2. **行宽**：不超过 80 个字符（嵌入式项目推荐）
3. **括号风格**：K&R 风格（开括号不换行）
   ```c
   void function_name(void) {
       // 代码
   }
   ```
4. **操作符空格**：二元操作符两侧加空格
   - 正确：`result = a + b * c;`
   - 错误：`result=a+b*c;`

5. **函数间间距**：函数定义之间用 2 个空行分隔

### 头文件规范
1. **防止重复包含**
   ```c
   #ifndef MODULE_NAME_H
   #define MODULE_NAME_H
   // 内容
   #endif // MODULE_NAME_H
   ```

2. **包含顺序**
   ```c
   // 系统头文件（尖括号）
   #include <stdint.h>
   #include <stdbool.h>
   
   // 项目头文件（双引号）
   #include "drv_canfd.h"
   #include "robstride_motor.h"
   ```

### 源文件结构
```c

 * @file module_name.c
```

## 嵌入式开发特定规范

### 内存管理
1. **避免动态内存分配**：嵌入式系统通常不使用 malloc/free
2. **使用静态分配**：预分配固定大小的缓冲区
3. **栈大小监控**：关注函数调用深度和局部变量大小

### 硬件操作
1. **寄存器访问**：使用 FSP 提供的 API，避免直接操作寄存器
2. **中断处理**：
   - ISR 函数尽可能简短
   - 避免在 ISR 中调用可能阻塞的函数
   - 使用 volatile 关键字修饰共享变量
3. **外设初始化**：遵循硬件手册的初始化序列

### 实时性要求
1. **任务优先级**：合理设置 FreeRTOS 任务优先级
2. **临界区保护**：使用互斥锁或信号量保护共享资源
3. **超时处理**：所有阻塞操作都应设置合理的超时时间

### 错误处理
1. **返回值检查**：所有函数调用都应检查返回值
2. **错误恢复**：提供错误恢复机制
3. **日志记录**：使用 sys_log 模块记录关键错误

## 模块开发指南

### CAN FD 模块
1. **消息定义**：在 `drv_canfd.h` 中统一消息 ID 和格式
2. **数据打包**：使用 `packet_packer.c` 工具进行数据序列化
3. **错误处理**：实现总线错误恢复机制

### WiFi 模块
1. **配置存储**：使用 `nvm_manager` 存储 WiFi 配置
2. **连接管理**：实现自动重连机制
3. **安全**：使用安全的连接方式（WPA2）

### 屏幕 UI 模块
1. **LVGL 集成**：遵循 LVGL 的事件驱动模型
2. **资源管理**：合理分配显示缓冲区
3. **响应式设计**：确保界面在不同状态下正确显示

### 语音识别模块
1. **模型集成**：正确集成 Edge Impulse 生成的模型
2. **音频处理**：确保音频采集和处理链路的实时性
3. **内存使用**：关注神经网络模型的内存占用

## 测试规范

### 单元测试
1. **核心算法**：对 PID 控制、滤波算法等进行单元测试
2. **硬件抽象**：对硬件驱动提供 mock 实现进行测试

### 集成测试
1. **模块间通信**：测试模块间的数据流
2. **系统功能**：测试完整的系统功能链

### 硬件在环测试
1. **实际硬件**：在目标硬件上运行测试
2. **边界条件**：测试极端条件下的系统行为

## 版本控制规范

### 分支策略
1. **主分支**：`main` 或 `master`，稳定版本
2. **开发分支**：`develop`，集成最新开发成果
3. **功能分支**：`feature/功能描述`，新功能开发
4. **修复分支**：`fix/问题描述`，Bug 修复

### 提交信息
```
类型: 简要描述

详细描述：
- 变更内容1
- 变更内容2
- 修复的问题

影响范围：
- 受影响的模块
```

**类型说明**：
- `feat:` 新功能
- `fix:` Bug 修复
- `docs:` 文档更新
- `style:` 代码格式（不影响功能）
- `refactor:` 代码重构
- `test:` 测试相关
- `chore:` 构建过程或辅助工具

## 安全规范

### 代码安全
1. **输入验证**：所有外部输入都应验证
2. **缓冲区安全**：确保不会发生缓冲区溢出
3. **权限控制**：根据功能需求最小权限原则

### 通信安全
1. **数据加密**：敏感数据应加密传输
2. **认证机制**：实现适当的身份认证
3. **防重放攻击**：重要操作防止重复执行

## 性能优化指南

### 代码层面
1. **循环优化**：减少循环内不必要的计算
2. **查表法**：复杂计算可预计算为查找表
3. **内联函数**：对小型频繁调用的函数使用 inline

### 系统层面
1. **任务调度**：合理设置任务优先级和周期
2. **内存布局**：优化数据结构和内存对齐
3. **外设使用**：合理配置外设时钟和功耗
